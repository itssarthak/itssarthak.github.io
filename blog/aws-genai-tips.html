Scaling AWS for GenAI: RAG Pipelines in Production

April 6, 2026 · 8 min read · By Sarthak Chhabra

Retrieval-Augmented Generation (RAG) has gone from a research curiosity to a production necessity in 2026. At Zupee, we built a RAG pipeline that handles over one million queries per day, and the journey taught us a lot about what works (and what doesn't) when you're running GenAI workloads on AWS at scale.

This guide covers the practical, battle-tested patterns we use. If you're building a RAG system with Node.js on AWS, these tips should save you weeks of trial and error.

Step 1: Setting Up S3 for Data Ingestion

Every RAG pipeline starts with data. At Zupee, our knowledge base consists of structured documents, FAQ databases, and real-time game rules. We store all raw documents in S3, organized by source and version.

The key insight is to treat S3 as your single source of truth. We use S3 event notifications to trigger Lambda functions whenever new documents are uploaded. The Lambda function handles parsing (PDF, JSON, HTML), chunking the text into semantically meaningful segments (typically 500-800 tokens), and pushing those chunks to our embedding pipeline.

For chunking, we found that overlapping windows (with 100-token overlap) significantly improve retrieval quality compared to naive splitting. We built a custom Node.js chunking service that respects paragraph boundaries and section headers.

Step 2: Vector Embeddings and Storage

Once documents are chunked, we generate vector embeddings using a managed embedding model via API calls. Each chunk gets a 1536-dimensional vector that captures its semantic meaning.

We store these vectors in a dedicated vector database alongside metadata (source document, chunk index, last modified date). The metadata is critical for filtering at query time — you often want to restrict retrieval to specific document categories or recency windows.

One optimization that made a huge difference: we pre-compute and cache embeddings for frequently asked queries. At Zupee, about 30% of incoming queries are variations of the same 500 questions. By caching the top-k retrieval results for these queries in ElastiCache (Redis), we eliminated the vector search entirely for a significant portion of traffic.

Step 3: Integrating RAG with the Application Layer

Our retrieval and generation pipeline is orchestrated through AWS Lambda functions written in TypeScript. The flow is straightforward:

1. The user query arrives via API Gateway. 2. A Lambda function generates the query embedding. 3. We perform a similarity search against our vector store (top-k=5). 4. Retrieved chunks are combined with the user query into a prompt. 5. The prompt is sent to the language model. 6. The response is streamed back to the client.

For step 2-4, latency is everything. We run these in a single Lambda function to avoid cross-function networking overhead. The Lambda is provisioned with 1024MB of memory (which also increases CPU allocation) and we use connection pooling for our vector database.

Case Study: Achieving 40% Latency Reduction

When we first launched the pipeline, our P95 latency was around 3.2 seconds. Users expect near-instant responses, especially in a gaming context. Here's what we did to bring it down to 1.9 seconds:

Caching layer: As mentioned, Redis caching for frequent queries eliminated vector search for ~30% of requests. This alone dropped average latency by 15%.

Embedding model optimization: We switched from a general-purpose embedding model to a smaller, domain-fine-tuned model. The vectors are slightly less "universal" but retrieval quality actually improved for our specific use case, and inference time dropped by 40%.

Provisioned concurrency: Cold starts were killing our P95 numbers. We use provisioned concurrency on the main Lambda functions to keep 10 warm instances available at all times. This costs more, but the latency improvement justified it.

Streaming responses: Instead of waiting for the full LLM response, we stream tokens back to the client using API Gateway WebSocket connections. The perceived latency drops dramatically because users see text appearing within 200ms of their query.

Monitoring and Observability

You cannot run a production RAG system without proper monitoring. We track several key metrics in CloudWatch: retrieval relevance scores, LLM response quality (via automated evaluation), end-to-end latency percentiles, cache hit rates, and embedding pipeline lag.

We also built a simple feedback loop: when users indicate a response was unhelpful, we log the query, retrieved chunks, and generated response. This data feeds into weekly quality reviews where we identify gaps in our knowledge base and fine-tune our chunking strategy.

Key Takeaways

Building RAG on AWS is not just about plugging models together. The real work is in data engineering (chunking, embeddings, caching), infrastructure optimization (Lambda sizing, provisioned concurrency, connection pooling), and operational excellence (monitoring, feedback loops, quality evaluation).

If you're starting a RAG project in 2026, invest heavily in your data pipeline before worrying about model selection. The quality of your retrieval step determines 80% of the output quality.
