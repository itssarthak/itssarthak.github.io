Node.js Microservices: Best Practices for Scalable Architectures

April 6, 2026 · 7 min read · By Sarthak Chhabra

Over the past six years, I've built and maintained microservice architectures at four different companies. The patterns that work at scale are surprisingly consistent, regardless of whether you're processing gaming transactions at Zupee or handling AI image pipelines at Dresma. Here are the practices I've found most valuable.

Architecture Overview: Event-Driven by Default

The single biggest architectural decision that pays off in microservices is committing to event-driven communication as the default. At Dresma AI, we used Kafka as our central event bus, and at Zupee we use a combination of AWS SQS and Kafka depending on the latency requirements.

The principle is simple: services communicate by publishing events, not by making synchronous HTTP calls to each other. When Service A completes an operation, it publishes an event to Kafka. Services B, C, and D each consume that event independently and react accordingly. This gives you loose coupling, natural retry semantics, and the ability to add new consumers without modifying the producer.

We use TypeScript for all our services. The type safety at service boundaries — especially for event schemas — prevents an enormous category of bugs that would otherwise only surface in production. We define shared event type definitions in a common package that all services depend on.

Service Boundaries: Domain-Driven Decomposition

The hardest part of microservices isn't the technology — it's deciding where to draw the boundaries. We follow Domain-Driven Design (DDD) principles: each service owns a bounded context with its own data store.

At Zupee, our core services include: User Service (authentication, profiles, preferences), Game Engine (match logic, scoring, state machines), Payment Service (transactions, wallet, payouts), and Notification Service (push, email, SMS). Each owns its database and never reads from another service's data store directly.

When services need data from each other, they either consume events to maintain local read-replicas (eventual consistency) or make explicit API calls through well-defined contracts. The local replica pattern is preferable for read-heavy scenarios because it eliminates runtime coupling.

Error Handling at Scale

In a distributed system, partial failures are the norm, not the exception. Every external call — database queries, API requests, Kafka publishes — can fail, and your service needs to handle that gracefully.

We follow three rules religiously:

Circuit breakers everywhere. We use a custom lightweight circuit breaker implementation in Node.js. When a downstream service starts failing, the circuit breaker trips after a configurable threshold (typically 5 failures in 30 seconds) and immediately returns an error for subsequent calls rather than waiting for timeouts. After a cooldown period, it lets a single request through to test if the service has recovered.

Idempotent operations. Every message consumer and API endpoint is designed to be safely retried. We use unique request IDs and database-level deduplication to ensure that processing the same event twice doesn't cause data corruption. This is non-negotiable when you're using at-least-once message delivery (which Kafka and SQS both provide).

Dead letter queues for everything. When a message fails processing after the configured retry count (we use 3 retries with exponential backoff), it goes to a dead letter queue. We have automated alerts on DLQ depth and a simple admin tool for replaying failed messages after the root cause is fixed.

Deployment with Docker

All our services are containerized with Docker and deployed to AWS ECS (Elastic Container Service). Each service has a Dockerfile that follows a multi-stage build pattern: the first stage compiles TypeScript and runs tests, the second stage copies only the compiled JavaScript and production dependencies into a minimal Node.js Alpine image.

This approach gives us images under 150MB, fast startup times (under 3 seconds), and consistent environments from development through production. We use Docker Compose locally so developers can spin up the entire service mesh with a single command.

Our CI/CD pipeline (Jenkins at Dresma, GitHub Actions at Zupee) runs on every PR: lint, type-check, unit tests, build Docker image, run integration tests against the containerized service, then push to ECR. Deployment to staging is automatic; production requires manual approval.

Observability: The Non-Negotiables

With dozens of services communicating asynchronously, you need three things to maintain sanity: structured logging with correlation IDs (we use Pino for high-performance JSON logging in Node.js), distributed tracing (AWS X-Ray), and metric dashboards (CloudWatch with custom metrics for business-level events).

The correlation ID pattern is especially important: every incoming request gets a unique trace ID that propagates through every downstream service call and Kafka event. When something goes wrong, you can reconstruct the entire request flow across services by searching for that single ID.

Key Takeaways

Microservices with Node.js and TypeScript can scale remarkably well if you invest in the fundamentals: event-driven communication, clear service boundaries, defensive error handling, containerized deployments, and thorough observability. The technology choices matter less than the architectural principles behind them.
